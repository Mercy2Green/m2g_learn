### 4. Existing Runnable Baselines

- **Qwen3-VL 2025:** four-mode visual reasoning。
- **RoboTwin 2.0 2025:** simulation generator feasibility。
- **Aff-Grasp 2025:** affordance-only input。
- **GR00T N1.7 current release:** policy/schema smoke test。

## B. What No Existing Work Directly Provides

- No public baseline directly produces `direct / use_tool / reposition / abort-recovery` for quadruped-arm tool-aware mobile manipulation.
- No public AIGC/sim pipeline directly generates tool/no-tool matched pairs for quadruped-arm mobile manipulation.
- No mobile manipulation VLA baseline explicitly evaluates tool necessity.
- No fixed-base tool-use baseline handles mobile base reposition as a first-class mode.
- GET-USE is the closest L3 evidence, but it is not runnable and does not cover quadruped or direct-vs-tool necessity.

## C. First-Stage Tests from Existing Baselines

### Test 1 — Qwen3-VL Reasoning Baseline

* **Source baseline:** Qwen3-VL, 2025
* **Intersection level:** Remove as evidence; runnable baseline only
* **Input:** simulation screenshot/multi-view images + instruction + optional tool-state fields
* **Output:** JSON with one of four modes and state judgments
* **Required modification:** prompt/schema; optional matched-pair few-shot or LoRA
* **What it tests:** mature VLM ability to reason over tool state and mode
* **What it cannot test:** robot action feasibility or mobile control
* **Runnable this week:** Yes

Conditions:

1. zero-shot；
2. independent few-shot；
3. matched-counterfactual few-shot；
4. wrong/shuffled state；
5. optional small-model LoRA only after prompting baseline。

### Test 2 — RoboTwin / MobileManiBench / Isaac Sim Data Feasibility

* **Source baseline:** RoboTwin 2.0 2025; MobileManiBench 2026 as schema reference
* **Intersection:** L1A / L2A
* **Input:** task, scene, robot and tool parameters
* **Output:** paired observations, states, actions and oracle mode
* **Required modification:** add direct feasibility、tool reachable/usable/suitable、base reposition feasibility
* **What it tests:** whether an existing simulation pipeline can produce auditable matched variants
* **What it cannot test:** learning benefit until connected to Qwen3-VL/policy
* **Runnable this week:** RoboTwin smoke test yes; quadruped extension uncertain

If existing project Isaac Sim is used:

> This is a project-specific extension inspired by 2025–2026 simulation baselines, not an existing tool-necessity baseline.

### Test 3 — Aff-Grasp Affordance-Only Baseline

* **Source baseline:** Aff-Grasp, ICCV 2025
* **Intersection:** L1C
* **Input:** RGB or predicted functional mask + instruction
* **Output:** affordance mask; Qwen3-VL converts it to four-mode answer
* **Required modification:** mask-to-prompt or overlay adapter
* **What it tests:** whether affordance alone explains decision gains
* **What it cannot test:** reachability, availability, necessity
* **Runnable this week:** Partial; code/data/weights available

### Test 4 — GR00T / π0.5 Schema Feasibility

* **Source baseline:** GR00T N1.7 current; π0.5/openpi 2025
* **Intersection:** L1A
* **Input:** images + state + language
* **Output:** continuous action chunks
* **Required modification:** custom embodiment/data config and later mode auxiliary target
* **What it tests:** whether project data can enter a current robot foundation policy
* **What it cannot test this week:** fair policy-level tool necessity
* **Runnable this week:** official demos only; not core

### Test 5 — Mobile State/Constraint Definition

* **Sources:** Human2LocoMan, SLIM, MoManipVLA, SG-VLA, RAKOMO
* **Input:** quadruped pose/proprioception, multi-view vision, arm state
* **Output:** state schema; later base-arm actions
* **Required modification:** define base/tool relative pose and reachability fields
* **What it tests:** not a model comparison; ensures the dataset reflects mobile embodiment
* **What it cannot test:** tool necessity independently
* **Runnable this week:** schema extraction only

## D. Data / Motivation Experiment

### Matched variables

- direct feasible ↔ infeasible；
- tool reachable ↔ unreachable；
- tool usable ↔ unusable；
- suitable tool present ↔ absent；
- current base pose feasible ↔ reposition required。

### Required controls

- same-count independent scenes；
- distance-matched hard pairs；
- shuffled/wrong tool state；
- affordance-only input；
- non-causal appearance changes；
- grouped split by base scene/tool instance。

### Metrics

- four-mode macro-F1；
- pairwise mode-flip consistency；
- direct/use-tool confusion；
- reposition/abort confusion；
- contradiction with supplied tool state；
- unnecessary-tool-use rate。

## E. Later AIGC Test

* **Sources:** RIGVid filtering, DreamGen generation, Dream2Fix verification, Video Policy/UWM future modeling。
* **Use:** visual/semantic variants only before action use。
* **Comparison:** sim original / generated unfiltered / generic filter / tool-state-label filter。
* **No claim:** generated video is reliable action supervision。

## F. Go / No-Go

**Go**

- matched Qwen3-VL condition improves mode-flip consistency；
- wrong-state degrades performance；
- gains persist on distance-matched pairs；
- simulation labels pass automatic feasibility audit。

**No-go**

- only random-split accuracy improves；
- affordance-only equals full state on all hard subsets；
- tool-state shuffling has no effect；
- AIGC changes pair labels frequently。

## G. Not to Do Now

- Do not train a new general VLM.
- Do not invent a custom classifier as the main baseline.
- Do not treat fixed-base tool papers as mobile baselines.
- Do not treat mobile manipulation papers as tool-aware evidence.
- Do not train UWM/Video Policy/DreamGen now.
- Do not use generated actions without verification.
- Do not build full WBC/trajectory or agent+skill systems.
