## 对核心问题的证据化回答

### Q1. 现有工作主要解决 how 还是 when？

**事实：** 工具论文的任务输入通常已经要求使用工具，或候选集合只包含工具；模型选择工具、抓取区域、接触配置和轨迹。RoboTool、CoPa、iTuP、Robustness-Aware Tool Selection 都属于这一模式。

**推断：** 主流是 “how/which tool”，不是 “whether a tool is necessary”。后者要求至少加入 direct/no-tool action，并对 unnecessary tool use 计成本。

### Q2. 是否已有明确 tool necessity 工作？

**事实：**

- RoboTool 包含“无工具不可行”的任务，但系统根据任务描述生成工具使用程序，不学习统一的 tool/no-tool necessity classifier。
- Learning at the Ends 比较 hand affordance 与 tool affordance，但重点是 affordance transfer。
- Creative Robot Tool Use by Counterfactual Reasoning 用物理属性反事实发现工具适用因果特征，但默认研究“哪个物体可当工具”。

**结论：** 截至检索日期，**没有找到形成成熟主线、标准 benchmark 和成对数据范式的 tool necessity 学习工作；证据不足**。不能声称绝对无人做过，但可以严谨地声称“系统化、可执行的 tool/no-tool necessity learning 仍缺少强代表工作”。

### Q3. 是否使用正负、失败、counterfactual scenes？

**事实：**

- SSDF 从失败轨迹中筛出可利用片段。
- AHA/FailGen 和 Guardian 通过扰动成功轨迹构造执行/规划失败。
- Dream2Fix 通过生成 world model 对动作做扰动，生成成对失败—修正数据。
- Creative Robot Tool Use by Counterfactual Reasoning 扰动工具几何/物理属性以发现因果特征。

**推断：** 技术积木已经存在，但尚未被系统用于 necessity decision。尤其缺少同一初始状态下 `direct success vs tool overhead`、`direct failure vs tool success`、`tool present but unusable` 的配对。

### Q4. AIGC / synthetic data 怎么被使用？

| Route | Examples | What is retained | What is generated | Typical use |
|---|---|---|---|---|
| Semantic visual augmentation | ROSIE, GenAug | 原动作/轨迹 | 物体、背景、干扰物、光照 | perception/generalization |
| Geometrically constrained transfer | EMMA | 多视角几何和动作 | 外观域、物体/背景变化 | VLA hybrid training |
| Generated visual plan | Gen2Act, RoboDreamer | 当前观测/语言 | 人类或机器人未来视频 | 提供高层 motion prior |
| Neural trajectories | DreamGen | 少量 embodiment data | robot video + pseudo actions | 扩展行为和环境 |
| Joint world/action model | UWM, Video Policy, GR-2 | video/action paired subset | future frames and/or actions | pretraining and policy |
| Generated failure | Dream2Fix | 真实成功示范 | action-perturbed failure rollout | detection/recovery |

### Q5. AIGC 数据的主要问题是什么？

优先级不是简单的图像质量排序：

1. **Action grounding**：视频动作无法可靠反演成机器人控制。
2. **Contact physics**：手/工具接触、摩擦、穿透、物体质量与受力不可信。
3. **Embodiment consistency**：机械臂关节、夹爪、工具几何在视频中漂移。
4. **Temporal/multi-view consistency**：遮挡后物体状态改变、跨视角不一致。
5. **Label leakage/semantic inconsistency**：生成结果改变了任务事实，却沿用原标签。
6. **Selection bias**：生成器更容易产生视觉顺滑的成功样本，覆盖不了关键 hard negatives。
7. **Sim/Gen-to-real gap**：纹理更真不代表动力学、传感噪声和控制延迟更真。

因此 Candidate 2 应筛“机器人决策有效性”，而不是只筛 FID、CLIP score 或人工美观。

### Q6. World model 是否学习“意图/何时行动”？

**事实：** RoboDreamer/Gen2Act 生成视觉计划；UWM、Video Policy、GR-2 联合或附加 action prediction；DreamGen 用 latent action/IDM 恢复动作。

**推断：** world model 首先学习的是条件视觉动态。只有满足以下条件，才能把收益归因到决策：

- 明确任务条件；
- 动作条件或可靠的 action decoder；
- 对“不同决策导致不同未来”的反事实分辨能力；
- 在 mode/necessity/action 指标上优于仅视觉预训练；
- 生成未来的质量与真实执行收益相关。

否则它可能只是更好的视觉表示或数据增强器。

### Q7. VLA 是否使用 tool state 改变 action？

**事实：** 通用 VLA 输入图像、语言和有时的 proprioception，理论上可隐式编码工具状态；affordance/工具论文则显式提供 mask、keypoint、工具配置或物理评分。

**证据缺口：** 很少有工作执行 tool-state intervention：

- 工具从可达改为不可达；
- 完好改为损坏；
- 合适改为尺寸不合适；
- 目标从 direct-reachable 改为 only-tool-reachable；
- 保持语言和其他视觉因素不变。

没有 intervention，就不能证明 action change 来自 tool state，而不是场景捷径。

### Q8. 四足臂新增 failure mode

| Failure mode | Why desktop-arm baselines miss it | Observable consequence |
|---|---|---|
| Base-arm-tool joint reachability | 固定基座无站位变量 | 工具拿到了但 tool tip 到不了 |
| Support/stability under contact | 固定基座近似刚性锚定 | 推/撬时机身倾斜、足端滑移 |
| Tool-induced collision envelope | 工具延长有效 body | 工具或手臂撞机身、环境、腿 |
| Locomotion-manipulation timing | 桌面任务无 gait phase | 走动振动造成抓取/接触误差 |
| Mode switching | 单臂通常只有 grasp/manipulate | direct、取工具、重定位、接触操作、放回工具 |
| Moving-view occlusion | 固定相机/基座较稳定 | 工具和目标频繁自遮挡 |
| Tool acquisition opportunity cost | 取工具可能需长距离移动 | 工具可用但不值得用 |
| WBC tracking mismatch | policy action 和真实 whole-body execution 分层 | 高层轨迹可行但底层无法稳定跟踪 |

### Q9. 哪些论文可作 baseline？

最直接：OpenVLA/Octo、RT-Affordance/AffordGrasp、MoManipVLA、UWM/Video Policy、AHA/FailGen、RoboTool/PLATO、UMI on Legs。具体实验见 `20260623_03_baselines_and_experiments.md`。

### Q10. 最小可行研究命题？

优先级：

1. Counterfactual tool necessity learning；
2. Robot-usefulness filtering for generated data；
3. Causal tool-state conditioning for VLA action/mode。

“训练全新 world model”“做大规模 benchmark”“完整四足臂系统”不适合作为第一篇最小命题。

# Part 4: 当前方向的关键 limitation

| Limitation | Papers showing it | Why important | Can motivate paper? |
|---|---|---|---|
| 1. 研究 how/which tool，不研究 whether | RoboTool, CoPa, iTuP, Robustness-Aware Tool Selection | 项目核心决策没有现成监督或 benchmark | **是，最强 motivation** |
| 2. 缺少 necessity 正负样本 | OpenVLA/Octo/π0.5 数据以成功示范为主；工具论文任务预设 | 模型容易把“看到工具”当成“应该用工具” | **是** |
| 3. 缺少同场景 counterfactual pair | AHA/Guardian 是执行扰动；Creative Tool Use 扰动物理属性 | 非配对数据难以隔离工具状态的因果贡献 | **是，方法和数据设计都可围绕它** |
| 4. AIGC 语义对但物理/action 错 | Gen2Act, DreamGen, UWM, Video Policy, EMMA | 工具接触任务对细微几何和动力学更敏感 | **是，但必须有下游 policy 证据** |
| 5. 视频预测不保证学到意图 | RoboDreamer, GR-2, UWM | 漂亮的未来帧可能不改善 necessity/mode | **是，适合做严格 ablation** |
| 6. Tool state 未被因果验证 | 通用 VLA；affordance-only methods | “tool-aware”很容易退化为视觉识别包装 | **是，适合 intervention benchmark** |
| 7. Sim/gen-to-real gap | AHA/FailGen, DreamGen, 四足 WBC works | 接触、摩擦、足端滑移和工具惯量难迁移 | **可作次级 motivation** |
| 8. Mobile embodiment 增加 reachability/stability/mode failures | MoManipVLA, UMI on Legs, RoboDuet | 单臂结论不能直接迁移到四足臂 | **是，但 embodiment 本身不是贡献** |

## Limitation 之间的逻辑链

```text
成功示范偏置
  -> 模型学会“任务词/工具外观 -> 用工具”
  -> 不知道不用工具是否更好，也不知道工具是否不可用
  -> 通用 VLA 即使成功，也无法证明依赖 tool state
  -> 需要同场景 counterfactual pairs + intervention evaluation
  -> 四足臂进一步要求 mode 与 base-arm trajectory 对状态变化作出可执行响应
```

# Part 5: 三个候选研究命题

## Candidate Claim 1

* **Main claim:** 成对的 tool/no-tool counterfactual supervision，比等量随机正负样本更有效地学习 tool necessity，并降低不必要用工具和漏用工具。
* **Problem / failure mode:** 模型根据任务词或工具出现做捷径判断；N0/N1 混淆。
* **Why existing methods fail:** 成功示范和工具预设任务没有 direct-action 对照；普通 failure perturbation 不改变策略层选择。
* **Proposed mechanism:** necessity head + mode-conditioned action head；对同一场景构造最小状态干预，使用 pairwise/contrastive loss 保证正确 mode flip。
* **Required data:** 仿真为主的 N0–N3 配对；少量真实校准；可选 AIGC 做外观变化但不生成核心物理标签。
* **Minimal experiment:** 3 类任务 × 40 基础场景 × 每场景 4 个 counterfactual；OpenVLA/Octo adapter 或轻量 VLM + waypoint head。
* **Key metrics:** macro-F1、N0/N1 error、unnecessary tool rate、counterfactual consistency、task success、tool acquisition cost。
* **Strongest objection:** 改变 reachability 等变量后标签太容易，模型只是看距离。
* **How to answer:** 加入匹配距离但不同工具状态、匹配视觉但不同动力学、不可见状态需 proprio/tool metadata 的 hard pairs；跨工具和跨布局测试。
* **Suitable venue tendency:** ICRA / IROS / RA-L；若跨 embodiment 和数据规模强，可投 CoRL。

## Candidate Claim 2

* **Main claim:** 对 AIGC 机器人视频，基于 kinematic/contact/action consistency 的筛选，比通用视觉质量筛选更能预测并提高下游 policy gain。
* **Problem / failure mode:** 生成视频视觉真实但工具穿透、接触错误、机械臂形态漂移、动作不可反演。
* **Why existing methods fail:** 多数用 CLIP/FID/人工质量或几何一致性；这些指标未针对 tool necessity/action validity。
* **Proposed mechanism:** 三层 verifier：语义状态机、2D/3D kinematic consistency、tool-object contact/action recoverability；按置信度加权或拒绝样本。
* **Required data:** 从真实/仿真种子轨迹生成多质量级视频；保留真实动作或用 IDM；人工标注小型 verification set。
* **Minimal experiment:** 固定同一 policy、同一 AIGC 数量，对比 no filter、CLIP filter、generic video-quality filter、proposed robot filter。
* **Key metrics:** verifier AUROC、accepted-data precision、pseudo-action error、policy success、真实收益与筛选分数相关性。
* **Strongest objection:** 过滤器只是把数据量减少，收益来自保守选择。
* **How to answer:** 控制接受数量；随机同量子集；soft weighting；按错误类型消融；报告 diversity。
* **Suitable venue tendency:** CoRL / ICRA；若数据和生成模型贡献足够强，可考虑 TRO/RA-L。

## Candidate Claim 3

* **Main claim:** 显式 tool state 与 necessity conditioning 会使 VLA 对工具可用性干预产生正确的 mode 和 trajectory 变化，并改善四足臂执行。
* **Problem / failure mode:** 普通 VLA 在工具损坏、不可达或不必要时仍重复训练分布中的工具动作。
* **Why existing methods fail:** tool state 只存在于像素；没有结构化状态、因果约束或 intervention 评测。
* **Proposed mechanism:** object/tool tokens 表示 pose、functional region、graspability、availability、tip、collision envelope；necessity latent 调制 mode/action decoder。
* **Required data:** 有结构化 tool state 的仿真轨迹、配对干预、少量真实 tool-state annotations。
* **Minimal experiment:** 比较 image-only VLA、image+tool mask、image+tool state、image+tool state+counterfactual loss；WBC 固定。
* **Key metrics:** mode flip accuracy、trajectory sensitivity、task success、collision、base travel、tool-state calibration。
* **Strongest objection:** 显式状态来自 oracle perception，真实系统不公平。
* **How to answer:** 分别报告 oracle、estimated、noisy state；把 perception error 与 policy error拆开；证明机制在噪声下仍有收益。
* **Suitable venue tendency:** CoRL / TRO / RA-L；若只做系统集成则更像 ICRA/IROS demo paper。

## 推荐选择

**先做 Candidate 1。** 它最小、最容易构造严格对照，也最直接回答项目独特问题。Candidate 2 可作为 Candidate 1 的数据支撑或后续独立论文；Candidate 3 在 Candidate 1 已证明 necessity signal 有价值后再做，否则容易成为“多一个输入 token”的工程结果。

## 哪个最可能只是工程 demo？

“Cosmos/Sora 生成视频 + 微调 VLA + WBC 上四足臂完成若干工具任务”最危险。若没有成对对照、机制消融、状态干预和跨任务泛化，它只能证明 pipeline 能跑，不能证明 tool-awareness。
